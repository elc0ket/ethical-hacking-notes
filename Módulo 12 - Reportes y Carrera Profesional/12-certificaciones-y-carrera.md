## Empezar en Bug Bounty

### **Paso 1: Preparación**

```
SKILLS NECESARIOS:
• Web hacking (OWASP Top 10) ⭐ ESENCIAL
• Burp Suite dominio
• Recon methodology
• Patience y persistencia

TIEMPO ESTIMADO:
• 3-6 meses de preparación
• 20+ horas/semana practicando
```

---

### **Paso 2: Elegir Programas**

```
PARA EMPEZAR:
• Programas con scope amplio
• Response time rápido
• Payouts claros
• Triagers activos

EVITAR AL INICIO:
• Programas muy competidos
• Out of scope complejo
• Sin payouts (solo reputation)

EJEMPLOS BUENOS:
• Programas de gobiernos
• Startups activas
• Medium-size companies
```

---

### **Paso 3: Metodología**

```
1. RECON (40% del tiempo)
   • Subdomain enumeration (Amass, Subfinder)
   • Port scanning
   • Technology fingerprinting
   • Archive.org (wayback)
   • JS files analysis

2. VULNERABILITY HUNTING (50%)
   • Burp Suite scanning
   • Manual testing
   • OWASP Top 10
   • Business logic flaws
   • IDORs, broken access control

3. REPORTING (10%)
   • Clear description
   • Impact explained
   • Steps to reproduce
   • Screenshots/videos
   • Remediation suggestion
```

---

### **Ejemplo de Reporte:**

````markdown
# Title: SQL Injection in search parameter

## Summary
The search functionality at /api/v1/search is vulnerable to 
blind SQL injection, allowing an attacker to extract sensitive 
database information.

## Steps to Reproduce
1. Navigate to https://target.com
2. Intercept search request
3. Inject payload: `' AND SLEEP(5)--`
4. Observe 5-second delay

## Proof of Concept
```http
GET /api/v1/search?q=test' AND SLEEP(5)-- HTTP/1.1
Host: target.com
````

Response time: 5.2 seconds

## Impact

- Complete database compromise
- User PII exposure
- Potential data modification/deletion

CVSS: 9.1 (Critical)

## Remediation

Use parameterized queries:

```python
cursor.execute("SELECT * FROM products WHERE name = ?", (query,))
```

## Supporting Material

[Attached: sqlmap output, screenshots]

## Tips de Bug Bounty

```

✅ HACER: • Leer scope completamente • Respetar out-of-scope • No hacer DoS • Un reporte = una vulnerabilidad • Seguir reporting guidelines • Ser paciente con triagers • Agregar value (remediation, impact)

❌ NO HACER: • Reportar duplicados sin leer • Spam de reportes low-quality • Violar términos del programa • Presionar por payout • Disclosure público antes de fix • Testing en producción agresivo

```
